## Story
Como um usuario do Churchify, eu quero que todos os dados gerados pela AI (episodios, analises, conteudo) sejam persistidos no banco de dados Supabase, para que meu trabalho nao seja perdido ao recarregar a pagina.

## Objetivo

Implementar o schema completo de banco de dados no Supabase (churches, episodes, analysis_results) com CRUD service layer, CHECK constraints, indexes, e garantir que todos os dados de AI gerados persistem no banco. Este e o debito mais critico do projeto — sem persistencia, nada funciona.

## Debitos Resolvidos

- DB-001: No data persistence — tudo perdido em refresh (28h)
- DB-013: No data validation at database level (2h) — CHECK constraints na criacao do schema
- DB-015: No migration strategy (0h) — resolvido automaticamente com Supabase migrations
- DB-016: analysisResult como JSONB sem indexes (0h) — GIN index incluido no schema
- SYS-012: No persistence para resultados de AI (deduplicado em DB-001)
- UX-005: Sem salvar/exportar resultados (deduplicado em DB-001)

### Key Technical Context
- **Zero persistence is the #1 critical debit** — currently all AI-generated data is lost on page refresh
- **Schema from DB-AUDIT.md:** churches (id, name, plan, logo_url, created_by, timestamps), episodes (id, church_id FK, title, youtube_url, sermon_date, status, analysis_result JSONB, created_by, timestamps)
- **church_members table (Task 12):** Required by Story 1.8 for RLS policies. Schema: church_id UUID FK -> churches.id, user_id UUID FK -> auth.users.id, role TEXT, joined_at TIMESTAMPTZ, PK(church_id, user_id). This junction table enables membership-based access control
- **Status enum:** CHECK constraint with values ('draft', 'processing', 'completed', 'failed')
- **GIN index:** On episodes.analysis_result for efficient JSONB queries (containment @>, existence ?, path operators)
- **Service layer pattern:** Use Supabase client from src/lib/supabase.ts, return typed responses
- **Supabase migrations:** Use `supabase migration new <name>` to create, `supabase db push` to apply
