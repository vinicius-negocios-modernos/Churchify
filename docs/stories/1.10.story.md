## Story
Como um administrador do Churchify, eu quero que a chave da API Gemini esteja protegida no servidor com rate limiting por usuario, para que a chave nao seja exposta no client bundle e o uso da API seja controlado.

## Objetivo

Mover a chave da API Gemini do client bundle para um backend proxy (Supabase Edge Function), implementar tracking de uso de API por usuario, e adicionar rate limiting basico. Isso elimina o risco critico de abuso da chave exposta.

## Debitos Resolvidos

- SYS-002: Gemini API key exposta em client bundle (6h)
- DB-010: No per-user API key management / usage tracking (8h)
- DB-017: No rate limiting at database level (4h) — rate limiting basico

### Key Technical Context
- **VULNERABILIDADE CRITICA (SYS-002):** `VITE_GEMINI_API_KEY` e incluida no bundle client-side via `import.meta.env` — qualquer usuario pode extrair do DevTools
- **Supabase Edge Functions** usam Deno runtime — `supabase.functions.invoke('generate-content', { body: {...} })` no client
- **Edge Function env:** A Gemini API key deve ser adicionada via `supabase secrets set GEMINI_API_KEY=xxx` (nao no .env do client)
- **Rate limiting:** Checar `SELECT count(*) FROM api_usage WHERE user_id = $1 AND created_at > now() - interval '24 hours'` antes de cada chamada
- **api_usage schema:** `id (uuid PK), user_id (FK profiles), tokens_used (int), model (text), endpoint (text), created_at (timestamptz default now())`
- **Dep Story 1.3:** Supabase project e CLI devem estar configurados — Edge Functions requerem projeto Supabase ativo e `supabase link`
- **Dep Story 1.7:** Schema base deve existir para FK em api_usage.user_id -> profiles.id

### Testing
- **Framework:** Vitest + React Testing Library (configurados em Story 1.2)
- **Edge Function tests:** Mock da Gemini API response; testar rate limiting com mock de api_usage queries
- **Security test:** `grep -r "API_KEY" dist/` deve retornar zero resultados apos implementacao
- **Localizacao de testes:** `src/__tests__/` ou co-located `*.test.ts`
