## Story
Como um administrador de igreja no Churchify, eu quero que os dados da minha igreja sejam completamente isolados de outras igrejas, com controle de membros por roles e audit trail, para que tenhamos seguranca e rastreabilidade em todas as operacoes.

## Objetivo

Implementar multi-tenancy completo: tabela church_members com roles (admin, editor, viewer), sistema de convites, isolamento total de dados por igreja via RLS refinado, e audit trail para operacoes senssiveis.

## Debitos Resolvidos

- DB-009: No multi-tenancy / church isolation (20h)
- DB-014: No audit trail / activity log (8h)
- SYS-016: Church name/plan hardcoded no sidebar (4h) — parcialmente resolvido em 1.11
- UX-007: Igreja/plano dinamico no sidebar (4h) — parcialmente resolvido em 1.11

### Key Technical Context
- **Supabase RLS:** Story 1.8 establishes base RLS policies. This story REFINES them with role-based granularity (viewer/editor/admin)
- **church_members table:** Junction table between `churches` and `profiles` (from Story 1.7 schema). Role enum: admin, editor, viewer
- **Audit trigger:** Use PostgreSQL trigger function to auto-log INSERT/UPDATE/DELETE on `episodes`, `church_members` tables into `audit_log`
- **Church switcher:** User may belong to multiple churches. Store selected church_id in localStorage/context. All queries filter by selected church_id
- **Isolation test pattern:** Create 2 test churches with different members. Verify Church A user cannot see Church B data via RLS
